> 把 `startPlan` / `pausePlan` / `resumePlan` / `cancelPlan` 从「只翻 status」变成「真执行 step」。
> 这是 v0.6.0 全功能（反馈循环 + 错误恢复 + retry/skip）3-4 周范围里的**最小可执行切片**。

## MVP 目标（v0.5.23 / v0.6.0-rc1）

让 `pilot plan run <id>` 真的能跑完一个 plan 写到磁盘的 TOML，并产出真实事件历史和 runtime snapshot。**sequential 策略**先，**3 个 action type 真执行**，剩下 stub + 标 unimplemented。

### 在范围内

- 新增 `core/plan-executor.ts`：
  - `class PlanExecutor` — 单 plan 实例，按 `startPlan` / `pausePlan` 触发
  - `async run()` — 顺序走 task × step，dispatch action
  - 持久化 `~/.pilot/runtime/plans/<id>.json`（每个 step 完成时落盘）
  - **可恢复**：进程挂掉后 `startPlan` 看 runtime snapshot，从断点继续
  - 每个 step 真实写 `step_started` / `step_completed` / `step_failed` 事件
  - 支持 pause（在 step 边界检查）+ resume + cancel（`AbortController` 切到底）
- 在 service-impl 里把 `startPlan` 从「只翻 status」改为「spawn executor + 走完」
  - 不阻塞 HTTP 响应（executor 是 fire-and-forget）
  - 或在 service-impl 启动时跑 recovery（扫 `runtime/plans/*.json` 找未完成的）
- 3 个 action type 真执行：
  - **`pilot_command`** → `child_process.spawn('pilot', [command, ...args], { cwd })` + 等 exit code
  - **`profile_switch`** → 调 `service.activateProfile(name)`
  - **`policy_apply`** → 调 `service.applyPolicy(name)`
- 其他 action type **stub**（不抛错，直接 success + warning 日志）：
  - `pi_session`、`pack_install`、`condition`、`wait`、`manual`
  - 标记 `data: { stubbed: true, reason: "v0.5.23 placeholder" }` 便于后续替换
- Tests：
  - 单元：`PlanExecutor` 用 fake services 跑一个 3-step 顺序 plan，断言 status / events / runtime
  - 集成：临时目录 + 真实 child_process 跑一个 `pilot_command: { command: "doctor" }` 的 plan
  - Pause/resume：mid-execution 暂停，断言 step 已完成 + status=paused + 可 resume
- 文档：CHANGELOG + roadmap 校准 (15)

### 不在范围内（v0.6.0 主版本才做）

- `pi_session` action 真执行（v0.5.14.3 的 bridge 还在生产可用性打磨阶段；先 stub）
- `pack_install` 真执行（pilot-tools 改造中等复杂度）
- `condition` / `wait` / `manual` 真分支
- `parallel` / `adaptive` strategy（保留 enum，sequential 真做）
- 重试 / 跳过 endpoint（`POST /plans/:id/tasks/:taskId/retry` / `skip`）
- step 实时 UI 推送（WebSocket push）— 当前保持轮询
- FeedbackEngine / 错误恢复策略
- 多个 plan 并发执行（先单 plan）

### 边界与假设

- **不阻塞 HTTP 响应**：`pilot plan run` / `POST /plans/:id/start` 立即返回 `{ status: "running" }`，executor 异步跑
- **崩溃恢复**：runtime snapshot 是单一事实源；service 启动时扫一遍 `runtime/plans/*.json`，status=`running` 的 plan 自动 resume
- **单进程**：MVP 不支持 multi-process lock；后续 v0.6.0 加 plan-level 文件锁
- **step 超时**：默认 5 分钟（`MAX_STEP_TIMEOUT_MS`），可被 `step.input.timeoutMs` 覆盖
- **cancellation 语义**：`cancel` 后正在跑的 step 等它自然完成，再翻 `cancelled`；不杀进程（避免脏状态）

## 关键设计

### PlanExecutor 内部状态机

```
            startPlan                every step
  draft ────────────▶ running ────────────────▶ (run loop)
                       │   ▲                       │
                       │   │ resume                │
                       │   └───── paused           │
                       │                           │
                       └──────── cancel ───────────┤
                                                   ▼
                          completed / failed / cancelled
```

每个 step 子状态：
```
pending ──run──▶ running ──ok──▶ completed
                       │
                       └─err──▶ failed   (no retry yet in MVP)
```

### 持久化格式（`runtime/plans/<id>.json`）

```json
{
  "planId": "20260711_xxx",
  "status": "running",
  "currentTaskId": "task_abc",
  "currentStepId": "step_xyz",
  "completedTaskIds": ["task_001"],
  "completedStepIds": ["task_001.step_a", "task_001.step_b"],
  "startedAt": "2026-07-11T00:00:00Z",
  "lastCheckpointAt": "2026-07-11T00:01:23Z"
}
```

每次 step 完成 → 落盘一次。崩溃 → restart → readPlanCore + runtime snapshot 算 diff。

## 风险

- **child_process 跑 pilot_command 时如果 plan 在 web UI 里触发**，pilot_command 会再 spawn 一个 pilot server（如果当前 server 还没起）— 不阻塞，但要 log warning
- **可恢复性 bug**：runtime snapshot 和 plan TOML 之间出现 drift（人为改 plan）→ MVP 直接 throw，提示用户重新跑
- **历史事件文件大小**：每个 step 一个 event JSONL；大 plan (100+ steps) 一次启动不卡，单次 append 也快（行追加）
